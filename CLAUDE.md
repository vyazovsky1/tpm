# AITPM Framework

## What This Repo Is
This is the AITPM (AI-First Technical Program Management) framework — a shared library of agent definitions, process descriptions, SDLC stage guides, and integration specs. It is framework-only: no program state, no program data lives here.

Each program using this framework gets its own folder generated by `scripts/new-program.sh`. That folder contains the program's per-stream state under `/streams`, an `integrations.md` registry, a `/.data` integration cache, and references this repo for all agent and process definitions.

## Directory Structure
```
agents/          — Agent definitions (one file per agent)
framework/       — SDLC stage files (planning, requirements, design, ...)
processes/       — Continuous management processes (knowledge, risk, comms, ...)
integrations/    — Integration specs with standard queries (Confluence, Drive, Gmail, ...)
program/         — Scaffold templates for generated programs:
                     templates/        — program-root files (CLAUDE.md, integrations.md, gitignore, data-README.md)
                     streams/_template/ — per-stream state, copied into each program's streams/<stream>/
scripts/         — Scaffold and auth scripts
docs/ideas/      — Design decisions and ideation artifacts
tasks/           — Implementation plan and todo tracker
```

## Generated Program Structure
A generated program folder has **no `/program` folder**. All live state lives under `streams/<stream>/`, one folder per stream (track / workstream); the set of subfolders under `streams/` is the stream registry. Each stream folder holds: `context.md`, `team.md`, `action-items.md`, `raid.md` (Risks, Assumptions, Issues, Dependencies), `decisions.md`, `knowledge.md`, `meetings/` (one summary file per meeting), `communications.md`, and `history/`. Risks, dependencies, issues, and action items are all per-stream; a cross-stream dependency or action item is duplicated in each affected stream with a reference to the other. Program-wide aggregation (e.g., a combined status report) is generated across streams, not stored. Agents operating in a stream read its `context.md` + `team.md` first. See `docs/ideas/stream-model.md`.

Two non-stream items live at the program root:
- **`integrations.md`** — the program's integration registry (markdown, committed). Each row binds a source (named for its framework spec / MCP server) to its scope and the stream it feeds. Edited by hand or by a script. No secrets — those stay in `.env`.
- **`/.data/<source>/`** — integration cache (gitignored, regenerable). One subfolder per source, named for the integration / MCP server (e.g. `.data/google-drive/`). Agents cache fetched data here and read it before fetching live; files may also be dropped here manually as an integration substitute.

## Role Names
Always capitalize role names: Engineering, Product, Stakeholders, QA, SRE, Security, Design, Management, Leadership.

## Conventions

### Agent Files (`agents/*.md`)
Each agent file must have these sections in order:
1. **Purpose** — one paragraph; what the agent does and what it does NOT do
2. **Data Sources** — table with columns: Source | What It Reads | What It Writes | Notes
3. **Triggers** — when the agent runs (scheduled / event-driven / on demand)
4. **Behaviors** — bullet list of core behaviors in plain language
5. **Agent Rules** — numbered rules; each rule is a named, precise instruction the agent follows
6. **Outputs** — what the agent produces
7. **Requires TPM Approval** — explicit list of actions that must not proceed without TPM sign-off

Mark out-of-MVP-scope items inline in the Data Sources Notes column: `Out of MVP scope`.

### Process Files (`processes/*.md`)
Structure: Purpose → Taxonomy or Model → TPM Responsibilities → Team Responsibilities → Agent Rules → AI Capabilities table (columns: Area | AI Role | How AI Helps | Agent).

### Framework Stage Files (`framework/*.md`)
Use `template.md` as the base. AI Capabilities table columns: Area | AI Role | How AI Helps | Agent. Mark out-of-MVP agents inline.

### Integration Files (`integrations/*.md`)
Each integration file must have: Purpose → Standard Queries (named queries with syntax) → Field Mapping table. Standard Queries are the exact calls agents make — keep them precise and copy-pasteable.

## MVP Scope
**Agents:** Program Brain, Knowledge Agent, Requirements Agent, Communications Agent, Meetings Agent  
**SDLC stages:** Planning, Requirements  
**Processes:** Knowledge Management, Communication Management  
**Integrations:** Confluence, Google Drive, Gmail, Google Chat, Meeting Notes

Everything else in this repo is defined but out of MVP scope. Out-of-MVP items are kept because they define the full framework shape; they are not executed until explicitly included.

## Adding a New Agent
1. Copy the structure from an existing agent file (e.g., `agents/knowledge-agent.md`)
2. Fill all seven sections
3. Mark any data sources not yet in scope as `Out of MVP scope` in the Notes column
4. If the agent is called by Program Brain, add a delegation rule to `agents/program-brain.md`
5. If the agent reads a stream state file, ensure that file exists in the per-stream template `program/streams/_template/`

## Adding a New Integration
1. Create `integrations/<service>.md` (the generic spec — `<service>` is the canonical source name)
2. Define: what data is available, standard query syntax for each use case, field mapping
3. Reference the integration in all agent files that use it
4. Add MCP server configuration to `.mcp.json` if needed
5. The source becomes available to programs as a row in their `integrations.md`, caching to `.data/<service>/` — keep the source name identical across the spec filename, the MCP server, the registry row, and the cache folder

## Scaffold Templates
All content the scaffolder writes into a generated program lives as real files under `program/` — **never hardcode template content inside `skills/init/scripts/init.sh`**. The script only copies templates and substitutes placeholders.

- **Program-root files:** `program/templates/` (`CLAUDE.md`, `integrations.md`, `gitignore`, `data-README.md`). Edit these to change what a new program's root looks like.
- **Per-stream files:** `program/streams/_template/`. Edit these to change a stream's starting state.
- **Placeholders:** `{{TOKEN}}` (mustache style), substituted by `init.sh` with `sed`. Root tokens: `{{PROGRAM_NAME}}`, `{{FRAMEWORK_DIR}}`, `{{STREAMS}}`. Stream token: `{{STREAM}}`. Literal `<source>` / `<stream>` angle-bracket placeholders in templates are intentional examples and are left untouched.

## Slash Commands
Slash commands live in each program folder at `.claude/commands/*.md`, not in this repo. They are generated by `scripts/new-program.sh` with this framework's absolute path baked in. To add a new command to all future program folders, add it to `scripts/new-program.sh`.

## Setting Up a New Program
```bash
bash scripts/new-program.sh <program-name> <target-path>
cd <target-path>
claude .
# Then run /tpm-kickoff inside Claude Code
```

## Google Authentication (MCP Servers)
MCP servers for Gmail, Google Drive, and Google Calendar require OAuth credentials.

1. Create a Google Cloud project and OAuth 2.0 client (Desktop app type)
2. Enable Gmail API, Drive API, Calendar API
3. Run: `node scripts/google-auth.js` — opens browser, saves refresh token to `.env`
4. MCP servers load credentials automatically via `scripts/mcp-google.sh`

Never commit `.env`. It is gitignored.

## TPM Decision Authority
Agents draft, detect, and surface. The TPM approves, decides, and acts. No agent updates `streams/<stream>/` state without explicit TPM approval. This is non-negotiable and must be enforced in every agent's `Requires TPM Approval` section.
